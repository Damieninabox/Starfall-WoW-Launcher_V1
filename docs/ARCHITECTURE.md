## 1. Tech Stack

**Tauri 2 + React + TypeScript + Tailwind.**

- Tauri can produce a single portable `.exe` via the bundler's `app` target (not `nsis`/`msi`). WebView2 is present by default on Windows 10 1803+ and Windows 11 — if a player is missing it, bootstrap silently on first run.
- Rust backend handles filesystem, process spawning, hashing, downloads, keychain.
- React frontend handles all UI. No shell-outs from JS.

Binary size target: under 20 MB for the launcher itself (patches/addons/client are downloaded on demand).

## 2. Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    Launcher.exe (Tauri)                      │
│  ┌───────────────────────┐    ┌──────────────────────────┐  │
│  │  React UI             │◄──►│  Rust Core               │  │
│  │  - Login              │IPC │  - File patcher          │  │
│  │  - Expansion picker   │    │  - Hash + verify         │  │
│  │  - Characters/Armory  │    │  - Download w/ resume    │  │
│  │  - M+ / Shop / News   │    │  - Realmlist R/W         │  │
│  │  - Settings           │    │  - Cache clear           │  │
│  └───────────┬───────────┘    │  - Spawn Wow.exe         │  │
└──────────────┼────────────────┴──────────┬──────────────────┘
               │ HTTPS                      │ Local FS / Process
               ▼                            ▼
     ┌──────────────────────────┐  ┌──────────────────────┐
     │ CMS API (account data)   │  │ Game Install         │
     │ + Patch host (Caddy +    │  │ %LOCALAPPDATA%/WoW   │
     │   static files)          │  │                      │
     └──────────────────────────┘  └──────────────────────┘
```

The patch host is a simple HTTPS static file server — Caddy recommended. Same box as the CMS or separate, doesn't matter to the launcher. See `PATCH_HOST_SETUP.md` for setup.

Install location default: `%LOCALAPPDATA%/<YourServerName>/` — picked because it doesn't need admin rights. Let players override.

## 3. Features

### 3.1 Expansion Picker (Cata 4.3.4 + placeholders)

UI: horizontal row of expansion cards (Classic, TBC, WotLK, Cata, MoP, retail-custom, whatever). Only Cata is clickable; others render greyed with a "Coming Soon" badge.

Config is data-driven so adding a new expansion later is a config change + asset drop, not a code change:

```json
{
  "expansions": [
    { "id": "classic", "name": "Classic", "version": "1.12.1", "enabled": false },
    { "id": "tbc",     "name": "The Burning Crusade", "version": "2.4.3", "enabled": false },
    { "id": "wotlk",   "name": "Wrath of the Lich King", "version": "3.3.5a", "enabled": false },
    {
      "id": "cata",
      "name": "Cataclysm",
      "version": "4.3.4",
      "enabled": true,
      "executable": "Wow.exe",
      "executable64": "Wow-64.exe",
      "realmlistPath": "Data/{locale}/realmlist.wtf",
      "configPath": "WTF/Config.wtf",
      "cachePaths": ["Cache", "WDB"],
      "manifestUrl": "https://patches.yoursite.com/manifests/cata.json"
    },
    { "id": "mop",     "name": "Mists of Pandaria", "version": "5.4.8", "enabled": false }
  ]
}
```

Serve this from the CMS (`GET /api/launcher/expansions`) rather than embedding it — adding an expansion later is then a zero-deploy change.

### 3.2 Unified Login

Single form: username + password → `POST /api/launcher/login` → JWT (15 min) + refresh token.

CMS-side, the endpoint checks credentials against whichever store is authoritative on your server (TrinityCore/AzerothCore SRP6 account table, website user table, or a unified view). That's the CMS's job, not the launcher's — the launcher just hits one endpoint.

Tokens stored in Windows Credential Manager via the `keyring` Rust crate. Never in JSON on disk.

"Remember me" = store refresh token. "Log out" = delete both.

No account creation in-launcher for v1 — link out to the website's register page.

**2FA (TOTP):** supported from day one. If the account has 2FA enabled, login flow shows a second screen for the 6-digit code. Setup flow (scan QR, enter code to confirm) in Settings → Security. Backup codes shown once, downloadable as `.txt`.

**Session history:** Settings → Security lists last N login events (timestamp, IP, country, user agent). Data from CMS. Suspicious entries get a "This wasn't me" button that force-logs-out all sessions and prompts password change.

### 3.3 Patcher (full client + patches + addons)

Biggest piece. Design once, right.

**Server-side manifest per expansion** — hosted on any HTTP server you control:

```json
{
  "expansion": "cata",
  "version": "2026.04.20",
  "files": [
    {
      "path": "Wow.exe",
      "size": 8675309,
      "sha256": "a1b2...",
      "url": "https://patches.yoursite.com/cata/Wow.exe",
      "category": "client"
    },
    {
      "path": "Data/enUS/patch-enUS-4.MPQ",
      "size": 104857600,
      "sha256": "c3d4...",
      "url": "https://patches.yoursite.com/cata/Data/enUS/patch-enUS-4.MPQ",
      "category": "patch"
    },
    {
      "path": "Interface/AddOns/ServerAnnounce/ServerAnnounce.toc",
      "size": 412,
      "sha256": "e5f6...",
      "url": "https://patches.yoursite.com/cata/Interface/AddOns/ServerAnnounce/ServerAnnounce.toc",
      "category": "addon"
    }
  ],
  "deletions": ["Interface/AddOns/DeprecatedAddon"]
}
```

**Patch host requirements** (whatever you pick must support):

- HTTPS with a valid cert (Let's Encrypt is fine; Caddy handles this automatically)
- HTTP `Range` requests (required for resume — test with `curl -r 0-1000 <url>`)
- Correct `Content-Length` on responses
- Ideally: can serve large files without timing out (Caddy/nginx defaults work; some shared hosts 504 on big downloads)

The URLs in the manifest don't have to point to the same host as the manifest itself — you can serve the manifest from your CMS and the large files from a separate file server if that's easier later.

**Manifest generation** is the missing piece most people forget. You need a script that:

1. Walks the client directory tree
2. Hashes every file (SHA-256)
3. Skips files you don't want to ship (e.g. old `WTF/`, `Logs/`, `Screenshots/`, `Errors/`)
4. Writes out `cata.json` in the shape above

Run it whenever files change on the server side. Ship this script in `tools/generate-manifest/` — it's Python or Node, 100 lines max. See `PATCH_HOST_SETUP.md` for a reference implementation.

**Launcher flow:**

1. Fetch manifest from `manifestUrl`
2. For each file: check exists → compare size (fast) → compare SHA-256 (slow, only if size matches)
3. Build work list: `download`, `replace`, `delete`
4. Progress UI: total bytes, current file, speed, ETA
5. Download in parallel (3–4 concurrent, configurable), HTTP Range header for resume
6. Verify hash after each download; mismatch → retry once → fail loudly
7. Atomic replace: download to `file.ext.partial`, rename on success

**Performance details that matter:**

- **Hash cache.** A 25GB install takes minutes to fully hash. Cache `{path → (mtime, size, sha256)}` in `%APPDATA%/<Launcher>/hash-cache.json`. Rehash only if mtime or size changed.
- **Don't hash on every start.** Check manifest version against last-verified version; if unchanged and cache is intact, skip verification. Offer a "Repair" button for forced rehash.
- **Resume.** Partial downloads must survive launcher restarts — track `.partial` files, resume via Range.
- **Bandwidth.** Speed limit setting. Don't saturate the pipe by default during normal Play-button flow; let "full install" saturate with consent.

**First run UX:** no install detected → "Install Cataclysm 4.3.4" → folder picker → download everything. Show size estimate before starting. Warn on pathological paths (`$RECYCLE.BIN`, OneDrive-synced, drive with < required free space).

**Addon manager:** addons are files in `Interface/AddOns/`. Same manifest, `category: "addon"`. Dedicated Settings → Addons page lists every server-shipped addon with: name, description, version, enabled toggle, and a "managed by launcher" badge. Disabled addons get their `.toc` renamed to `.toc.disabled` so WoW ignores them without the patcher immediately re-downloading. Player-installed addons in the same folder are left alone (detect by: not in the manifest). Default: all server addons sync-on.

### 3.4 Realmlist

Cata's realmlist: `Data/<locale>/realmlist.wtf` with:
```
set realmlist logon.yourserver.com
```

- On first run post-install, write the default
- On every launch, read and compare to configured server — if different, prompt (don't silently overwrite; some players run test realms)
- Profile structure for future-proofing (Main / PTR / Event), but v1 ships with one

Write as UTF-8 no BOM, LF line endings. Some 4.3.4 builds are picky.

### 3.5 Cache Clear

On launch, delete contents of `Cache/` and `WDB/`. No WTF wipes. Verify target path is inside the configured install dir before any delete.

Setting: `on-launch` (default) | `weekly` | `manual-only` | `off`.

### 3.6 Character List + Armory

`GET /api/account/characters` → list, grouped by realm, sorted by last-played.

Click character → armory panel. For v1, embed the website's armory page in a Tauri WebView with an SSO token. Native React rebuild can come later if iframe feels janky.

Cache character list for 60s.

### 3.7 Competitive & Progression

Read-only endpoints from the CMS:

- `GET /api/mplus/affixes/current` → current week's affixes + icons
- `GET /api/mplus/leaderboard?dungeon=X&limit=100`
- `GET /api/mplus/character/:guid/runs`
- `GET /api/raids/progression` → `[{ raid, boss, killed, firstKillGuild, firstKillDate }]`
- `GET /api/raids/character/:guid` → per-character raid progress

**Mythic+ tab:** affixes card on home screen (high visibility; players check weekly), full leaderboard on dedicated tab, per-character runs on armory view.

**Raid progression tab:** grid of current tier bosses × top guilds showing kill status and dates. "Server first" banner for the guild that downed end-boss first. Per-character progress shown on armory view alongside M+ score.

### 3.8 Donation Shop

WebView tab pointed at `https://yoursite.com/shop?embedded=launcher&token=<sso>`. CMS hides site chrome when `embedded=launcher`. "Open in browser" button for users who prefer their payment autofill.

No payment code inside the launcher.

### 3.9 Polish (Phase 1, all cheap)

- **News feed** — `GET /api/launcher/news` → home screen
- **Server status** — `GET /api/launcher/server-status` → `{ online, population, uptime }` widget, poll every 30s
- **Launch options** — checkboxes for `-windowed`, `-console`, custom args textbox
- **Dark/light theme** — respect OS preference, toggle in settings
- **Dynamic backgrounds** — each expansion card has its own background art (Deathwing for Cata, etc.). Home screen uses the selected expansion's art. Art is local (shipped with launcher) to avoid flicker. Seasonal overlays (Hallow's End, Winter Veil) swapped via CMS flag.

### 3.10 Guild MOTD + Calendar

Small widget on the home screen pulls the player's guild info:
- `GET /api/guild/:guid` → `{ name, motd, memberCount, onlineCount }`
- `GET /api/guild/:guid/events?upcoming=true` → upcoming raid/event entries

Read-only in v1 — no creating/editing events from the launcher. Click-through opens the website's guild page for management.

### 3.11 Friend Referral

Settings → Refer a Friend shows:
- Player's unique referral link (`https://yoursite.com/r/<code>`)
- Count of signups and active referrals
- Rewards earned / pending

Endpoints:
- `GET /api/account/referral` → `{ code, link, signups, active, rewardsEarned }`

Reward logic lives on the CMS; launcher just displays.

### 3.12 Transmog Wishlist / Item Hunter

Dedicated tab. Players add items to a wishlist; launcher shows where each drops and what the player is missing.

- `GET /api/items/search?q=...` → item search
- `GET /api/account/wishlist` → items on wishlist
- `POST /api/account/wishlist` → add item
- `DELETE /api/account/wishlist/:itemId`
- `GET /api/items/:id/sources` → `[{ type: boss|quest|vendor, name, zone, dropChance? }]`

For the "collection progress" view (T11 Retribution: 5/8), the CMS needs a mapping of set pieces → set name. That's a content data concern, not launcher code.

### 3.13 Bug Report

Help → Report a Bug form: title, category (game/launcher/website), description, optional screenshot. On submit:
1. Launcher auto-attaches: `WowError.txt` (last one), launcher version, OS build, recent launcher log entries (last 100 lines)
2. POST to `/api/support/tickets` with the payload
3. Show ticket ID + link to CMS support page

All attached data is listed to the player before submit so there are no surprises about what's being sent.

## 8. Notable Risks

| | |
|---|---|
| **SmartScreen flags unsigned exe** | Scary warning on first download. Document the "More info → Run anyway" path on your website. Fixable later with a cert. |
| **Self-hosted patch bandwidth** | 25GB × N players hits your server's upstream hard. A single full install can saturate a typical 100Mbit upstream for ~30 min. Mitigations: offer client download as a torrent in parallel (free, offloads bandwidth), cap concurrent downloads per IP at the nginx level, or revisit a CDN later if it becomes a real problem. |
| **Partial downloads corrupt install** | `.partial` staging + atomic rename. Never write directly to target path. |
| **Player picks OneDrive folder** | Detect and warn. OneDrive sync will fight the patcher. |
| **4.3.4 realmlist.wtf encoding quirks** | Test with a real client before shipping. |
| **First-run full install is 25GB** | Set expectations in UI: size, ETA, "you can leave this running" message. |
| **Patch host missing Range support** | Resume breaks silently. Test with `curl -r 0-1000 <url>` before wiring the launcher to production. |
